# 📄 오픈소스 소프트웨어(OSS) 발전과 역사 GitHub 기능 활용 보고서


# 1. 오픈소스 소프트웨어(OSS) 개요

오픈소스 소프트웨어(Open Source Software, OSS)는 소스코드를 누구나 열람하고, 수정하고, 재배포할 수 있도록 공개된 소프트웨어를 의미한다.
기존의 상용 소프트웨어가 기업 중심으로 폐쇄적으로 개발되었다면, OSS는 전 세계 개발자들이 자유롭게 참여할 수 있는 개방형 개발 모델을 기반으로 한다.

OSS는 단순히 “무료로 쓸 수 있다”는 개념을 넘어 지식 공유, 협업, 투명성, 혁신 속도를 핵심 가치로 삼는다.
이러한 가치 때문에 오늘날 기업·기관·학교·개발자 개인 등 거의 모든 기술 생태계에서 OSS가 필수 요소가 되었다.

또한 OSS는 사용자가 버그를 직접 수정하거나 기능을 추가해 프로젝트에 기여할 수 있어 높은 확장성과 지속성도 가진다.
대표적으로 리눅스 커널은 수천 명의 개발자들이 참여하며 빠르게 진화하고 있으며, 기업 또한 아마존, 구글, 마이크로소프트와 같은 빅테크 기업들까지 OSS 개발을 적극적으로 후원하고 있다.

# 2. OSS의 발전 역사

오픈소스는 단순히 기술의 변화가 아니라, “개발 문화” 자체의 변화였다. 아래는 주요 변천 과정이다.

● 1970~1980: 자유 소프트웨어 운동(FSF)의 출발

리처드 스톨먼은 소프트웨어가 점점 폐쇄화되는 상황에 반발하여 “Free Software(자유 소프트웨어)” 개념을 제시했다.
그는 소프트웨어의 4대 자유(사용·연구·수정·재배포) 를 주장했고, 이를 위해 GNU 프로젝트를 시작했다.

● 1998: “Open Source” 용어 등장 및 OSI 설립

‘자유(Free)’라는 단어가 오해를 불러일으키고 기업에 부정적 이미지를 주는 문제를 보완하기 위해 “Open Source”라는 새로운 용어가 만들어졌다.
Open Source Initiative(OSI)가 설립되면서 공식 인증 체계가 생기고, 기업들이 오픈소스를 적극 도입하기 시작했다.

● 2000년대: 인터넷 & 커뮤니티 기반 개발 확대

인터넷의 보급으로 개발자 간 협업이 쉬워졌고, 포럼·메일링 리스트·버그트래커 등이 활성화되며 OSS는 하나의 거대한 생태계로 성장했다.

● 2005~현재: Git과 GitHub 등장으로 대규모 협업 혁명

리누스 토발즈가 만든 Git(2005) 은 코드 버전 관리를 혁명적으로 변화시켰고, 이어 GitHub(2008) 이 출시되면서 전 세계 개발자들이 한곳에서 프로젝트를 관리하고 기여할 수 있게 되었다.

현재 대부분의 인프라, 모바일 OS, 클라우드, 웹 서비스가 OSS 기반으로 운영되고 있으며, 정부·기업·교육기관 모두 OSS를 전략적으로 활용하는 시대가 되었다.

# 3. 대표적인 오픈소스 소프트웨어
● Linux

서버·AI·클라우드 등 거의 모든 인프라의 기반 운영체제.
기업용 서버의 90% 이상이 리눅스를 바탕으로 한다.

● Python

데이터 분석, 머신러닝, 웹 개발, 자동화 모든 분야에 걸쳐 사용되며, 풍부한 라이브러리 생태계로 빠르게 성장했다.

● Firefox

개인정보 보호와 오픈웹 철학을 강조하는 브라우저.
웹 표준을 지키는 대표적인 OSS 프로젝트이다.

● Android

전 세계 스마트폰 OS의 대부분을 차지하고 있으며, 모바일 생태계 발전에 큰 기여를 했다.

# 4. OSS 라이선스(저작권)

오픈소스는 아무 제한 없이 사용하는 것이 아니라, 프로젝트마다 준수해야 할 허용 범위가 명확하게 작성되어 있다.

● GPL

소스코드 공개를 요구하는 가장 강력한 카피레프트 라이선스

이 라이선스를 사용하는 프로그램은 수정·배포 시 소스코드를 반드시 공개해야 한다

● MIT

거의 모든 제약이 없는 자유로운 라이선스

상업적 사용 가능, 출처만 명시하면 됨

● Apache 2.0

기업 친화적이며 특허 보호 조항이 포함됨

대규모 프로젝트, 기업 OSS에서 많이 선택

● BSD

최소 제한 조건

학계·연구용 라이브러리에 많이 사용

라이선스 준수는 개발 윤리이자 OSS 생태계 유지의 기본이다.

# 5. GitHub 기능 활용 및 팀 프로젝트 평가

이번 프로젝트에서 GitHub의 다양한 기능을 활용하여 OSS 개발 방식과 협업 시스템을 직접 경험했다.

● 1) Issue (이슈)

팀원 간 해야 할 작업을 명확하게 분리하고 우선순위 지정에 도움

버그, 질문, 작업 요청을 체계적으로 관리할 수 있었음

● 2) Labels (레이블)
이슈에 색상과 분류 태그를 붙여 종류와 성격을 구분하는 기능임.
레이블 덕분에 이슈를 한눈에 파악하고 우선순위를 정하기 쉬워졌음.


● 3) Milestones (마일스톤)

프로젝트 전체 목표를 기간별로 구분하여 진행 상황을 확인

마감일까지 해야 할 업무가 정리되어 팀 일정 관리에 도움이 됨

● 4) Pull Request (PR)
작성한 코드를 메인 브랜치에 병합하기 전 리뷰하고 수정하는 과정임.
PR을 통해 코드 품질을 높이고 팀원 간 피드백이 자연스럽게 이루어졌음.

● 5) Wiki

팀 규칙, 프로젝트 개요, 기술 문서 등을 정리하는 공간

프로젝트 이해도 향상에 매우 유용했음

● 6) Projects(프로젝트 보드)
이슈와 할 일을 보드 형태로 구성해 진행 단계를 시각적으로 관리하는 기능임.
프로젝트 보드를 통해 작업 현황을 쉽게 파악하고 일정 관리가 효율적이었음.
